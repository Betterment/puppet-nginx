The GitHub Stack

We don't have any hard and fast rules for which technologies to use. But we really like Ruby, Mustache, SCSS, Coffeescript, UIKit, twui, and WPF.

Make sure other people are on board before introducing a new technology stack by asking around or posting an Idea to Team.

Building/Improving a Feature

Every feature on GitHub can be improved. Create a branch, code up an improvement you want to make, and ask for feedback. When you're ready to ship:

W

Write a blog post draft.

q

Create a pull request containing the code to ship it to public.

M

Write an email to all@github.com asking for feedback with links to the blog post and pull request.

t

Wait a little while for feedback/bugs (and fix them). The waiting period is subjective - Issues 3.0 needs more than 24 hours for everyone to test it, but a UI update might only need a day. It's your call. Get some awesome feedback from fresh eyeballs before shipping to the masses.

G

Ship it!

k

Own it. If bugs pop up, fix them.

There Will Be Bugs

It's everyone's responsibility to fix bugs. Every one of our repositories has a list of bugs under the "bug" label. If you want to know the most important bugs to fix, ask the Support team. They're the final word on bug priorities.

If you're looking to learn a new project, take a look at the bug list first. Try and fix a bug in a branch and submit a pull request.

If you run into any administrative problems stopping you from fixing a bug (tests won't pass, can't get the app running) grab your buddy and fix the issue. A broken environment is not an excuse to not fix bugs.

If you have questions about the best way to approach a technical problem, ask @rtomayko. If you have questions about the best way to approach a design problem, ask @kneath.

Absolutely any work toward fixing a bug is helpful. Do some work, send a pull request. Grammar correction, design updates from sysadmins, ruby patches from designers - it's everyone's job to ensure our products are stable and error free.
